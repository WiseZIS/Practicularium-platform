# ORDER-LIFECYCLE-01
## Жизненный цикл заказа

**Версия:** 1.0

**Статус:** Draft

**Проект:** PRACTICULARIUM Platform 2.0

---

# 1. Назначение документа

Данный документ определяет полный жизненный цикл заказа (Order Lifecycle) в системе PRACTICULARIUM OMS.

Документ является основой для:

- работы Order Management System (OMS);
- построения бизнес-процесса обработки заказов;
- автоматизации изменения статусов;
- интеграции с платежными системами;
- интеграции с сервисами доставки;
- автоматической отправки уведомлений клиентам;
- формирования событий заказа (Order Events);
- дальнейшего развития Workflow Engine.

---

# 2. Основные принципы Order Lifecycle

Жизненный цикл заказа (Order Lifecycle) определяет единые правила прохождения заказа через все этапы обработки в рамках платформы PRACTICULARIUM.

Настоящий раздел устанавливает основные архитектурные принципы, обязательные для всех модулей платформы независимо от способа продажи, типа товара, способа оплаты или способа исполнения заказа.

Все последующие разделы документа должны соответствовать данным принципам.

---

## Принцип 1. Статус отражает только текущее состояние заказа

Каждый статус отвечает только на один вопрос:

> **В каком состоянии находится заказ в настоящий момент?**

Статусы не используются для описания ожиданий, внутренних процессов системы или отдельных действий пользователей.

---

## Принцип 2. Один заказ — один активный статус

В каждый момент времени заказ может находиться только в одном статусе.

При изменении статуса предыдущий статус считается завершённым и сохраняется только в истории заказа.

---

## Принцип 3. История заказа не является частью статуса

Все действия, происходящие во время нахождения заказа в определённом статусе, фиксируются отдельно в журнале событий (Order Events).

Изменение статуса и регистрация событий являются независимыми процессами.

---

## Принцип 4. OMS является единственным источником истины

Окончательное состояние заказа определяется исключительно системой Order Management System (OMS).

Внешние сервисы не изменяют состояние заказа напрямую.

Они только передают события, на основании которых OMS принимает решение о возможном изменении статуса.

---

## Принцип 5. Бизнес определяет архитектуру

Жизненный цикл заказа строится исходя из реального бизнес-процесса издательства.

Программная реализация не должна изменять утверждённую бизнес-логику.

При необходимости изменяется программный код, а не бизнес-процесс.

---

## Принцип 6. Универсальность жизненного цикла

Настоящий документ описывает полный перечень возможных состояний заказа.

Конкретный заказ использует только те статусы, которые необходимы выбранному сценарию исполнения (Fulfillment Scenario).

Отсутствие отдельных статусов в конкретном сценарии не считается нарушением жизненного цикла.

---

## Принцип 7. Автоматизация не изменяет бизнес-процесс

Автоматизация предназначена исключительно для выполнения утверждённых бизнес-правил.

Автоматические действия системы не создают новых статусов и не изменяют утверждённую последовательность жизненного цикла заказа.

---

## Принцип 8. Масштабируемость платформы

Жизненный цикл заказа проектируется как универсальная модель, пригодная для обработки любых видов продукции и способов исполнения заказа.

Добавление новых товаров, способов оплаты, служб доставки или цифровых сервисов не должно требовать изменения базовой архитектуры Order Lifecycle.



---



# 3. Универсальный словарь статусов

Настоящий раздел определяет полный перечень статусов, которые могут использоваться платформой PRACTICULARIUM.

Данный перечень является единым для всех модулей системы.

Каждый статус представляет собой самостоятельную контрольную точку бизнес-процесса.

Конкретный сценарий исполнения заказа (Fulfillment Scenario) использует только необходимый набор статусов из настоящего словаря.

---

## Основные статусы жизненного цикла

| Статус | Наименование | Назначение |
|---------|--------------|------------|
| NEW | Новый заказ | Заказ успешно зарегистрирован системой |
| CONFIRMED | Подтверждён | Заказ проверен менеджером и принят в обработку |
| PAID | Оплачен | Получено подтверждение успешной оплаты |
| PREPARING | Подготовка | Заказ находится в процессе подготовки к исполнению |
| READY_FOR_PICKUP | Готов к самовывозу | Заказ полностью подготовлен и ожидает получения покупателем |
| MEETING_SCHEDULED | Назначена встреча | Согласованы место и время личной передачи заказа |
| SHIPPED | Отправлен | Заказ передан службе доставки |
| DIGITAL_DELIVERY | Цифровая доставка | Электронный товар передан покупателю |
| HANDED_OVER | Передан лично | Заказ передан покупателю без участия службы доставки |
| DELIVERED | Доставлен | Перевозчик подтвердил успешную доставку заказа |
| FUNDS_RECEIVED | Средства получены | Денежные средства окончательно поступили на расчётный счёт ФОП |
| COMPLETED | Завершён | Бизнес-процесс полностью завершён |

---

## Исключительные статусы

| Статус | Наименование | Назначение |
|---------|--------------|------------|
| CANCELLED | Отменён | Заказ отменён до завершения бизнес-процесса |
| NOT_RECEIVED | Не получен | Покупатель не получил заказ в установленный срок хранения |
| RETURNED | Возвращён | Заказ возвращён продавцу |
| REFUNDED | Возврат средств | Денежные средства полностью возвращены покупателю |

---

## Архитектурные правила использования статусов

### Каждый статус имеет одно значение

Один статус может описывать только одно состояние заказа.

Объединение нескольких состояний в одном статусе не допускается.

---

### Статусы не дублируются

В платформе отсутствуют статусы, описывающие ожидание выполнения действия.

Например:

- ожидание оплаты;
- ожидание отправки;
- ожидание получения.

Подобные процессы контролируются автоматическими бизнес-правилами OMS и не являются отдельными статусами.

---

### Использование статусов определяется сценарием исполнения

Каждый Fulfillment Scenario самостоятельно определяет, какие статусы используются в процессе исполнения заказа.

Поэтому отсутствие отдельного статуса в конкретной цепочке не считается нарушением архитектуры платформы.

---

### Расширение словаря

Добавление новых статусов допускается только при изменении бизнес-процесса платформы.

Добавление нового способа доставки, нового товара или нового платёжного сервиса само по себе не является основанием для создания новых статусов.


---

## 4.1 NEW

### Назначение

Статус **NEW** присваивается заказу автоматически сразу после его успешного оформления на сайте.

На данном этапе заказ считается зарегистрированным в системе, но ещё не проверенным менеджером и не принятым в обработку.

Статус **NEW** является первой контрольной точкой жизненного цикла заказа.

---

### Когда устанавливается

Статус устанавливается автоматически после успешного выполнения следующих действий:

- покупатель корректно заполнил форму заказа;
- данные успешно прошли проверку (валидацию);
- заказ успешно сохранён в базе данных;
- заказ получил уникальный номер;
- покупателю отправлено письмо с подтверждением получения заказа;
- администратору отправлено уведомление о новом заказе.

Только после успешного выполнения всех перечисленных действий заказ считается успешно зарегистрированным.

---

### Кто устанавливает

Автоматически.

Ответственные подсистемы:

- OMS;
- Create Order Service.

Участие менеджера на данном этапе не требуется.

---

### Автоматические действия системы

После установки статуса **NEW** OMS автоматически выполняет следующие действия:

1. Генерирует уникальный номер заказа.

2. Создаёт запись в базе данных.

3. Присваивает заказу статус **NEW**.

4. Фиксирует дату и время регистрации заказа.

5. Отправляет покупателю письмо о регистрации заказа.

6. Отправляет администратору уведомление о новом заказе.

7. Создаёт первое событие в журнале заказа (Order Events).

---

### Контрольная точка

После получения статуса **NEW** система гарантирует, что:

- заказ успешно зарегистрирован;
- информация о заказе сохранена;
- покупатель получил подтверждение регистрации заказа;
- администратор уведомлён о новом заказе;
- заказ готов к проверке менеджером.

---

### Следующий допустимый статус

```
CONFIRMED
```

---

### Запрещённые переходы

Из статуса **NEW** запрещён прямой переход в следующие статусы:

- PAID;
- PREPARING;
- READY_FOR_PICKUP;
- MEETING_SCHEDULED;
- SHIPPED;
- DIGITAL_DELIVERY;
- HANDED_OVER;
- DELIVERED;
- FUNDS_RECEIVED;
- COMPLETED.

Переход к следующим этапам возможен только после проверки заказа менеджером.

---

### Ответственный

OMS.

После завершения стадии **NEW** ответственность за дальнейшую обработку заказа переходит менеджеру OMS.

---

### Примечание

Статус **NEW** является обязательным для всех сценариев исполнения заказа (Fulfillment Scenarios), реализованных на платформе PRACTICULARIUM.

Любой заказ независимо от типа товара, способа доставки или способа исполнения начинается именно со статуса **NEW**.


---


## 4.2 CONFIRMED

### Назначение

Статус **CONFIRMED** означает, что заказ полностью проверен менеджером и официально принят издательством к исполнению.

На данном этапе менеджер подтверждает возможность выполнения заказа на заявленных условиях.

После установки статуса **CONFIRMED** OMS автоматически информирует покупателя о принятии заказа в работу и предоставляет инструкции для его оплаты.

Статус **CONFIRMED** является первой управляемой контрольной точкой жизненного цикла заказа.

---

### Когда устанавливается

Статус устанавливается после проверки менеджером следующих условий:

- корректность персональных данных покупателя;
- наличие выбранного товара;
- корректность количества;
- корректность способа исполнения заказа (Fulfillment Scenario);
- корректность выбранного способа оплаты;
- корректность стоимости заказа;
- возможность выполнения заказа издательством.

Только после успешной проверки всех перечисленных условий заказ может быть переведён в статус **CONFIRMED**.

---

### Кто устанавливает

На текущем этапе развития платформы:

Менеджер OMS.

В перспективе допускается частичная автоматизация процесса подтверждения заказа.

---

### Автоматические действия системы

После установки статуса **CONFIRMED** OMS автоматически выполняет следующие действия:

1. Сохраняет новый статус заказа.

2. Фиксирует дату и время подтверждения заказа.

3. Создаёт событие в журнале заказа (Order Events).

4. Формирует письмо покупателю.

5. Указывает окончательную стоимость заказа.

6. Добавляет доступные способы оплаты.

7. Формирует ссылку на оплату (при использовании онлайн-платежей).

8. Указывает срок действия ссылки на оплату (если применимо).

9. Отправляет письмо покупателю.

10. Запускает автоматический контроль оплаты заказа.

---

### Контрольная точка

После получения статуса **CONFIRMED** система гарантирует, что:

- заказ принят издательством;
- выполнение заказа подтверждено;
- стоимость окончательно согласована;
- покупатель получил инструкции по оплате;
- OMS начинает автоматический контроль дальнейшего прохождения заказа.

---

### Следующий допустимый статус

```
PAID
```

---

### Запрещённые переходы

Из статуса **CONFIRMED** запрещён прямой переход в следующие статусы:

- PREPARING;
- READY_FOR_PICKUP;
- MEETING_SCHEDULED;
- SHIPPED;
- DIGITAL_DELIVERY;
- HANDED_OVER;
- DELIVERED;
- FUNDS_RECEIVED;
- COMPLETED.

До подтверждения оплаты дальнейшее исполнение заказа невозможно.

---

### Ответственный

Менеджер OMS.

После подтверждения заказа OMS автоматически сопровождает дальнейшее прохождение заказа в соответствии с утверждёнными бизнес-правилами.

---

### Примечание

Статус **CONFIRMED** является обязательным для всех сценариев исполнения заказа (Fulfillment Scenarios).

Период ожидания оплаты не выделяется в отдельный статус.

Контроль сроков оплаты, отправка напоминаний покупателю и уведомлений менеджеру выполняются автоматическими бизнес-правилами OMS и фиксируются в журнале событий (Order Events), не изменяя текущего статуса заказа.



---


## 4.3 PAID

### Назначение

Статус **PAID** означает, что OMS получила официальное подтверждение успешного поступления денежных средств по данному заказу.

После получения данного статуса заказ считается полностью оплаченным и допускается к дальнейшему исполнению.

Статус **PAID** является первой финансовой контрольной точкой жизненного цикла заказа.

---

### Когда устанавливается

Статус устанавливается после получения подтверждения успешной оплаты заказа.

На текущем этапе развития платформы подтверждение оплаты выполняется менеджером OMS.

В дальнейшем изменение данного статуса должно выполняться автоматически после получения подтверждения от интегрированных финансовых сервисов.

---

### Кто устанавливает

На текущем этапе:

Менеджер OMS.

В дальнейшем:

Автоматически OMS после получения подтверждения оплаты от:

- Checkbox;
- банковской системы;
- платёжного шлюза;
- иной утверждённой финансовой интеграции.

---

### Автоматические действия системы

После установки статуса **PAID** OMS автоматически выполняет следующие действия:

1. Сохраняет новый статус заказа.

2. Фиксирует дату и время подтверждения оплаты.

3. Создаёт событие в журнале заказа (Order Events).

4. Разрешает переход к этапу исполнения заказа.

5. Подготавливает данные для дальнейших интеграций.

6. Активирует следующий этап жизненного цикла.

---

### Контрольная точка

После получения статуса **PAID** система гарантирует, что:

- денежные средства подтверждены;
- заказ допущен к исполнению;
- OMS может начать подготовку заказа;
- дальнейшая обработка заказа разрешена.

---

### Следующий допустимый статус

Следующий статус определяется выбранным сценарием исполнения заказа (Fulfillment Scenario).

Например:

- PREPARING;
- DIGITAL_DELIVERY;
- MEETING_SCHEDULED.

---

### Запрещённые переходы

Из статуса **PAID** запрещён прямой переход в следующие статусы:

- SHIPPED;
- DELIVERED;
- HANDED_OVER;
- READY_FOR_PICKUP;
- FUNDS_RECEIVED;
- COMPLETED.

До выполнения соответствующего сценария исполнения переход к данным статусам невозможен.

---

### Ответственный

На текущем этапе:

Менеджер OMS.

В дальнейшем:

OMS автоматически после получения подтверждения оплаты от финансовых сервисов.

---

### Примечание

Статус **PAID** является обязательным для всех сценариев исполнения заказа.

Момент формирования фискального чека не определяется данным статусом.

Алгоритм формирования, регистрации и отправки фискального чека определяется отдельным документом архитектуры платформы, посвящённым интеграции с ПРРО (Checkbox), и реализуется в соответствии с требованиями действующего законодательства.

## 4.4 PREPARING

### Назначение

Статус **PREPARING** означает, что заказ успешно оплачен и издательство приступило к подготовке товара к исполнению выбранного сценария заказа.

На данном этапе OMS контролирует выполнение внутренних операций издательства, необходимых перед передачей товара покупателю или службе доставки.

Статус **PREPARING** используется только для товаров, требующих физической подготовки.

---

### Когда устанавливается

Статус устанавливается после подтверждения успешной оплаты заказа.

Переход в данный статус означает официальное начало подготовки заказа к исполнению.

---

### Кто устанавливает

На текущем этапе развития платформы:

Менеджер OMS.

В дальнейшем допускается автоматический переход после подтверждения оплаты, если это допускается выбранным сценарием исполнения заказа.

---

### Автоматические действия системы

После установки статуса **PREPARING** OMS автоматически выполняет следующие действия:

1. Сохраняет новый статус заказа.

2. Фиксирует дату и время начала подготовки.

3. Создаёт событие в журнале заказа (Order Events).

4. Разрешает выполнение внутренних операций по подготовке заказа.

5. Подготавливает данные для следующего этапа выбранного сценария исполнения.

6. Активирует внутренний контроль готовности заказа.

---

### Действия менеджера

На данном этапе менеджер выполняет необходимые внутренние операции издательства.

В зависимости от вида товара это может включать:

- подготовку экземпляра книги;
- проверку качества товара;
- авторскую подпись экземпляра (при необходимости);
- вложение закладки;
- вложение открытки;
- комплектацию заказа;
- подготовку сопроводительных документов.

Конкретный перечень операций определяется внутренними регламентами издательства и не влияет на жизненный цикл заказа.

---

### Контрольная точка

После получения статуса **PREPARING** система гарантирует, что:

- заказ полностью оплачен;
- исполнение заказа официально начато;
- товар находится в подготовке;
- OMS готова к переходу на следующий этап выбранного сценария исполнения.

---

### Следующий допустимый статус

Следующий статус определяется выбранным сценарием исполнения заказа.

Например:

- READY_FOR_PICKUP;
- SHIPPED;
- HANDED_OVER.

---

### Запрещённые переходы

Из статуса **PREPARING** запрещён прямой переход в следующие статусы:

- DELIVERED;
- FUNDS_RECEIVED;
- COMPLETED.

Переход к данным статусам возможен только после выполнения соответствующего сценария исполнения заказа.

---

### Ответственный

Менеджер OMS.

В дальнейшем часть внутренних операций может выполняться автоматически посредством интеграции с внешними сервисами.

---

### Примечание

Статус **PREPARING** используется только для физических товаров.

Для электронных товаров данный статус отсутствует.

Использование статуса определяется выбранным сценарием исполнения заказа (Fulfillment Scenario).




---



## Статусы исполнения заказа

Следующие статусы используются только в тех сценариях исполнения заказа (Fulfillment Scenarios), где их применение предусмотрено бизнес-процессом.

Не каждый заказ проходит через все перечисленные ниже статусы.

Использование конкретного статуса определяется выбранным сценарием исполнения заказа и типом реализуемого товара.

Отсутствие отдельного статуса в конкретном сценарии не считается нарушением жизненного цикла заказа.

Все последующие статусы настоящего раздела относятся исключительно к этапу исполнения заказа.




---


## 4.5 READY_FOR_PICKUP

### Назначение

Статус **READY_FOR_PICKUP** означает, что заказ полностью подготовлен и ожидает получения покупателем непосредственно в издательстве.

Данный статус используется только для сценария исполнения **Самовывоз**.

---

### Когда устанавливается

Статус устанавливается после завершения подготовки заказа.

Переход в данный статус означает, что заказ полностью готов к выдаче покупателю.

---

### Кто устанавливает

На текущем этапе развития платформы:

Менеджер OMS.

В дальнейшем допускается автоматическое изменение статуса после завершения всех внутренних операций подготовки.

---

### Автоматические действия системы

После установки статуса **READY_FOR_PICKUP** OMS автоматически выполняет следующие действия:

1. Сохраняет новый статус заказа.

2. Фиксирует дату и время готовности заказа.

3. Создаёт событие в журнале заказа (Order Events).

4. Формирует уведомление покупателю.

5. Отправляет информацию о готовности заказа к получению.

6. Запускает контроль срока хранения подготовленного заказа.

---

### Контрольная точка

После получения статуса **READY_FOR_PICKUP** система гарантирует, что:

- заказ полностью подготовлен;
- покупатель уведомлён;
- заказ ожидает получения;
- OMS контролирует срок хранения заказа.

---

### Следующий допустимый статус

```
COMPLETED
```

---

### Запрещённые переходы

Из статуса **READY_FOR_PICKUP** запрещён переход в статусы:

- SHIPPED;
- DELIVERED;
- DIGITAL_DELIVERY;
- HANDED_OVER;
- FUNDS_RECEIVED.

---

### Ответственный

Менеджер OMS.

---

### Примечание

Статус используется исключительно для сценария исполнения **Самовывоз**.

При превышении установленного срока хранения OMS может автоматически создать напоминание менеджеру или покупателю.

Правила контроля сроков хранения определяются отдельным регламентом платформы.



---


## 4.6 MEETING_SCHEDULED

### Назначение

Статус **MEETING_SCHEDULED** означает, что заказ полностью подготовлен, а покупатель и представитель издательства согласовали место и время личной передачи заказа.

Данный статус используется исключительно для сценария исполнения **Личная передача**.

---

### Когда устанавливается

Статус устанавливается после:

- подтверждения оплаты заказа;
- подготовки товара к передаче;
- согласования даты, времени и места встречи.

Переход в данный статус означает готовность обеих сторон к выполнению личной передачи заказа.

---

### Кто устанавливает

На текущем этапе развития платформы:

Менеджер OMS.

В дальнейшем допускается автоматическое создание напоминаний и контроль согласованной встречи посредством Reminder Engine.

---

### Автоматические действия системы

После установки статуса **MEETING_SCHEDULED** OMS автоматически выполняет следующие действия:

1. Сохраняет новый статус заказа.

2. Фиксирует дату и время согласования встречи.

3. Создаёт событие в журнале заказа (Order Events).

4. Отправляет покупателю подтверждение согласованной встречи.

5. Активирует контроль наступления времени встречи.

6. При необходимости использует механизмы Reminder Engine для автоматического напоминания покупателю и менеджеру.

---

### Контрольная точка

После получения статуса **MEETING_SCHEDULED** система гарантирует, что:

- товар полностью подготовлен;
- встреча согласована;
- обе стороны уведомлены;
- OMS контролирует наступление времени передачи заказа.

---

### Следующий допустимый статус

```
HANDED_OVER
```

---

### Запрещённые переходы

Из статуса **MEETING_SCHEDULED** запрещён переход в статусы:

- SHIPPED;
- DELIVERED;
- DIGITAL_DELIVERY;
- READY_FOR_PICKUP;
- FUNDS_RECEIVED.

---

### Ответственный

Менеджер OMS.

---

### Примечание

Статус используется исключительно для сценария исполнения **Личная передача**.

При переносе времени встречи статус не изменяется.

Изменения фиксируются в журнале событий (Order Events).

## 4.7 SHIPPED

### Назначение

Статус **SHIPPED** означает, что заказ официально передан службе доставки и принят перевозчиком к перевозке.

С момента установки данного статуса ответственность за физическое перемещение отправления переходит к выбранной службе доставки.

Статус **SHIPPED** используется исключительно для сценариев исполнения, предусматривающих доставку через перевозчика.

---

### Когда устанавливается

Статус устанавливается после подтверждения перевозчиком факта принятия отправления.

На текущем этапе проекта изменение статуса выполняется менеджером OMS.

После реализации интеграции с API служб доставки изменение статуса должно происходить автоматически после получения соответствующего подтверждения от перевозчика.

---

### Кто устанавливает

На текущем этапе:

Менеджер OMS.

После реализации Sprint 7:

Автоматически OMS после получения подтверждения от API службы доставки.

---

### Автоматические действия системы

После установки статуса **SHIPPED** OMS автоматически выполняет следующие действия:

1. Сохраняет новый статус заказа.

2. Фиксирует дату и время передачи отправления перевозчику.

3. Создаёт событие в журнале заказа (Order Events).

4. Сохраняет транспортную накладную (TTN), если она предусмотрена выбранной службой доставки.

5. Отправляет покупателю уведомление об отправке заказа.

6. Передаёт покупателю номер транспортной накладной и ссылку для отслеживания отправления.

7. Запускает автоматический контроль изменения статусов доставки.

8. При необходимости использует механизмы Reminder Engine.

---

### Контрольная точка

После получения статуса **SHIPPED** система гарантирует, что:

- заказ передан перевозчику;
- перевозчик подтвердил принятие отправления;
- транспортная накладная зарегистрирована (при наличии);
- покупатель получил информацию для отслеживания доставки;
- OMS начинает автоматический мониторинг доставки.

---

### Следующий допустимый статус

```
DELIVERED
```

---

### Запрещённые переходы

Из статуса **SHIPPED** запрещён переход в следующие статусы:

- READY_FOR_PICKUP;
- MEETING_SCHEDULED;
- DIGITAL_DELIVERY;
- HANDED_OVER;
- COMPLETED.

Переход возможен только после подтверждения факта доставки либо обработки исключительной ситуации.

---

### Ответственный

На текущем этапе:

Менеджер OMS.

После реализации интеграции:

OMS совместно с API службы доставки.

---

### Примечание

Статус **SHIPPED** используется только для сценариев исполнения, предусматривающих участие службы доставки.

Все изменения этапов транспортировки фиксируются в журнале событий (Order Events).

Статус **SHIPPED** не изменяется при прохождении промежуточных этапов перевозки.

Информация о сортировке, транзите, прибытии в отделение и других этапах доставки относится к журналу событий заказа и не является отдельными статусами жизненного цикла.



---


## 4.8 DELIVERED

### Назначение

Статус **DELIVERED** означает, что покупатель успешно получил заказ, а факт передачи товара подтверждён выбранным сценарием исполнения заказа.

Для сценариев с участием службы доставки данный статус устанавливается после получения официального подтверждения перевозчика.

Для других сценариев исполнения данный статус не используется.

---

### Когда устанавливается

Статус устанавливается после получения подтверждения успешной передачи товара покупателю.

Источником подтверждения могут являться:

- API службы доставки;
- подтверждение менеджера OMS;
- иной подтверждённый источник информации, предусмотренный сценарием исполнения заказа.

---

### Кто устанавливает

На текущем этапе:

Менеджер OMS.

После реализации интеграции со службами доставки:

Автоматически OMS после получения подтверждения от API перевозчика.

---

### Автоматические действия системы

После установки статуса **DELIVERED** OMS автоматически выполняет следующие действия:

1. Сохраняет новый статус заказа.

2. Фиксирует дату и время получения заказа покупателем.

3. Создаёт событие в журнале заказа (Order Events).

4. Завершает автоматический мониторинг доставки.

5. Отправляет покупателю уведомление об успешном завершении доставки (при необходимости).

6. Передаёт управление следующему этапу жизненного цикла заказа.

7. При необходимости активирует механизмы Reminder Engine.

---

### Контрольная точка

После получения статуса **DELIVERED** система гарантирует, что:

- товар получен покупателем;
- перевозчик завершил выполнение своих обязательств;
- OMS завершила контроль процесса доставки;
- заказ готов к переходу на следующий этап жизненного цикла.

---

### Следующий допустимый статус

Следующий статус определяется выбранным сценарием исполнения заказа.

Например:

```
FUNDS_RECEIVED
```

или

```
COMPLETED
```

---

### Запрещённые переходы

Из статуса **DELIVERED** запрещён переход в следующие статусы:

- PREPARING;
- READY_FOR_PICKUP;
- MEETING_SCHEDULED;
- SHIPPED;
- DIGITAL_DELIVERY;
- HANDED_OVER.

Возврат к предыдущим этапам исполнения заказа не допускается.

---

### Ответственный

На текущем этапе:

Менеджер OMS.

После реализации интеграции:

OMS совместно с API службы доставки.

---

### Примечание

Статус **DELIVERED** используется только для сценариев исполнения, предусматривающих участие службы доставки.

После подтверждения доставки OMS завершает контроль логистического этапа и передаёт управление следующему этапу бизнес-процесса.

Все изменения, происходящие после доставки (например, получение наложенного платежа, возврат товара или иные события), фиксируются отдельными статусами либо журналом событий заказа.




---



## 4.9 DIGITAL_DELIVERY

### Назначение

Статус **DIGITAL_DELIVERY** означает, что электронный товар успешно передан покупателю посредством цифрового канала доставки.

Данный статус используется исключительно для цифровых товаров и заменяет этапы физической доставки.

После установки данного статуса OMS считает обязательства издательства по передаче цифрового товара выполненными.

---

### Когда устанавливается

Статус устанавливается после успешной передачи покупателю цифрового товара.

Передача может осуществляться посредством:

- персональной ссылки для скачивания;
- защищённого кабинета пользователя;
- электронной почты;
- иного утверждённого цифрового канала распространения.

---

### Кто устанавливает

На текущем этапе развития платформы:

OMS автоматически.

При необходимости допускается подтверждение менеджером OMS.

---

### Автоматические действия системы

После установки статуса **DIGITAL_DELIVERY** OMS автоматически выполняет следующие действия:

1. Сохраняет новый статус заказа.

2. Фиксирует дату и время передачи цифрового товара.

3. Создаёт событие в журнале заказа (Order Events).

4. Формирует индивидуальную ссылку доступа (при необходимости).

5. Передаёт покупателю электронный товар.

6. Отправляет уведомление об успешной передаче.

7. Фиксирует результат доставки.

---

### Контрольная точка

После получения статуса **DIGITAL_DELIVERY** система гарантирует, что:

- цифровой товар успешно передан покупателю;
- обязательства издательства по передаче товара выполнены;
- OMS завершила контроль исполнения заказа.

---

### Следующий допустимый статус

```
COMPLETED
```

---

### Запрещённые переходы

Из статуса **DIGITAL_DELIVERY** запрещён переход в следующие статусы:

- PREPARING;
- READY_FOR_PICKUP;
- MEETING_SCHEDULED;
- SHIPPED;
- HANDED_OVER;
- DELIVERED;
- FUNDS_RECEIVED.

---

### Ответственный

OMS.

При необходимости — менеджер OMS.

---

### Примечание

Статус **DIGITAL_DELIVERY** используется исключительно для цифровых товаров.

Передача цифрового товара подтверждается средствами самой платформы.

При повторной выдаче доступа статус заказа не изменяется.

Все повторные отправки, восстановление ссылок доступа и другие действия фиксируются исключительно в журнале событий заказа (Order Events).

# Финансовое завершение заказа

Следующие статусы используются после завершения исполнения заказа.

Они отражают завершение финансовой части сделки между издательством и покупателем.

Не каждый сценарий исполнения заказа использует все статусы данного раздела.

Использование конкретного статуса определяется выбранным способом оплаты и сценарием исполнения заказа.

## 4.10 FUNDS_RECEIVED

### Назначение

Статус **FUNDS_RECEIVED** означает, что денежные средства по заказу окончательно поступили на расчётный счёт ФОП.

Данный статус используется только в тех сценариях, где между передачей товара покупателю и поступлением денежных средств существует временной разрыв.

Наиболее типичным примером является наложенный платёж.

---

### Когда устанавливается

Статус устанавливается после получения официального подтверждения поступления денежных средств.

Источником подтверждения могут являться:

- банковская интеграция;
- API финансового сервиса;
- платёжная система;
- подтверждение менеджера OMS.

---

### Кто устанавливает

На текущем этапе развития платформы:

Менеджер OMS.

После реализации финансовых интеграций:

OMS автоматически.

---

### Автоматические действия системы

После установки статуса **FUNDS_RECEIVED** OMS автоматически выполняет следующие действия:

1. Сохраняет новый статус заказа.

2. Фиксирует дату и время поступления денежных средств.

3. Создаёт событие в журнале заказа (Order Events).

4. Завершает финансовый контроль заказа.

5. При необходимости использует механизмы Reminder Engine для контроля задержки поступления денежных средств.

---

### Контрольная точка

После получения статуса **FUNDS_RECEIVED** система гарантирует, что:

- денежные средства поступили на расчётный счёт ФОП;
- финансовые обязательства покупателя полностью исполнены;
- OMS завершила контроль поступления денежных средств;
- заказ готов к окончательному закрытию.

---

### Следующий допустимый статус

```
COMPLETED
```

---

### Запрещённые переходы

Из статуса **FUNDS_RECEIVED** запрещён переход в любые предыдущие статусы исполнения заказа.

Переход возможен только к статусу **COMPLETED** либо к исключительным статусам в случае возникновения внештатной ситуации.

---

### Ответственный

На текущем этапе:

Менеджер OMS.

После реализации финансовых интеграций:

OMS совместно с банковскими и платёжными сервисами.

---

### Примечание

Статус **FUNDS_RECEIVED** используется только для тех сценариев исполнения заказа, где поступление денежных средств происходит после передачи товара покупателю.

Для заказов с полной предварительной оплатой данный статус не используется.




---



## 4.11 COMPLETED

### Назначение

Статус **COMPLETED** означает полное завершение жизненного цикла заказа.

Все обязательства между издательством и покупателем исполнены.

Заказ считается закрытым.

---

### Когда устанавливается

Статус устанавливается после успешного завершения последнего этапа выбранного сценария исполнения заказа.

Например:

- после поступления денежных средств (наложенный платёж);
- после передачи цифрового товара;
- после получения заказа при полной предоплате;
- после личной передачи товара;
- после завершения самовывоза.

---

### Кто устанавливает

На текущем этапе:

Менеджер OMS.

В дальнейшем:

OMS автоматически после выполнения всех условий завершения заказа.

---

### Автоматические действия системы

После установки статуса **COMPLETED** OMS автоматически выполняет следующие действия:

1. Сохраняет новый статус заказа.

2. Фиксирует дату и время полного завершения заказа.

3. Создаёт заключительное событие в журнале заказа (Order Events).

4. Закрывает Workflow заказа.

5. Завершает работу Reminder Engine по данному заказу.

6. Освобождает заказ от активного мониторинга OMS.

7. Передаёт заказ в раздел завершённых заказов.

8. Подготавливает данные для аналитики и отчётности.

---

### Контрольная точка

После получения статуса **COMPLETED** система гарантирует, что:

- жизненный цикл заказа полностью завершён;
- все обязательства сторон исполнены;
- OMS завершила сопровождение заказа;
- заказ доступен только для просмотра, аналитики и формирования отчётности.

---

### Следующий допустимый статус

Отсутствует.

Статус **COMPLETED** является конечной точкой жизненного цикла заказа.

---

### Запрещённые переходы

После получения статуса **COMPLETED** переход в любые рабочие статусы жизненного цикла заказа запрещён.

Изменения допускаются только при возникновении исключительных ситуаций, предусмотренных отдельными бизнес-регламентами платформы.

---

### Ответственный

OMS.

---

### Примечание

Статус **COMPLETED** является обязательной завершающей стадией любого успешно выполненного заказа независимо от выбранного сценария исполнения.

После завершения заказа OMS прекращает выполнение автоматических бизнес-процессов, связанных с данным заказом.

Дальнейшие действия (например, повторная покупка, обращение в поддержку, анализ продаж, маркетинговые кампании или программа лояльности) относятся к другим подсистемам платформы и не являются частью жизненного цикла заказа.

# 5. Fulfillment Scenarios

## Назначение раздела

Настоящий раздел определяет типовые сценарии исполнения заказов (Fulfillment Scenarios), поддерживаемые платформой PRACTICULARIUM.

Каждый сценарий представляет собой последовательность статусов жизненного цикла заказа, используемую для исполнения конкретного способа передачи товара покупателю.

Все сценарии используют единый словарь статусов, определённый в настоящем документе.

Различие между сценариями заключается исключительно в составе используемых статусов и порядке их прохождения.

---

## Общие правила

Каждый заказ использует только один сценарий исполнения.

Сценарий определяется автоматически на основании выбранного покупателем способа получения товара и типа приобретаемого продукта.

Изменение сценария исполнения после подтверждения заказа допускается только менеджером OMS при наличии соответствующих оснований и фиксируется в журнале событий заказа (Order Events).

Все сценарии подчиняются единым архитектурным принципам платформы и используют единый Workflow Engine.

Использование различных сценариев исполнения не изменяет общую архитектуру Order Lifecycle и не требует создания отдельных бизнес-процессов.


---


## 5.1 Fulfillment Scenario — Нова Пошта

### Назначение

Настоящий сценарий используется для исполнения заказов, доставляемых посредством сервиса доставки **«Нова Пошта»**.

Сценарий применяется для физических товаров и предусматривает полный контроль жизненного цикла заказа со стороны OMS до момента полного завершения сделки.

География доставки, доступные страны, отделения, почтоматы, способы доставки, ограничения, сроки и тарифы определяются Delivery Provider посредством API.

OMS использует исключительно актуальные данные, полученные от интегрированного сервиса, и не хранит собственные локальные справочники указанных данных.

---

### Используемые статусы

```text
NEW
 ↓
CONFIRMED
 ↓
PAID
 ↓
PREPARING
 ↓
SHIPPED
 ↓
DELIVERED
 ↓
FUNDS_RECEIVED
 ↓
COMPLETED
```

---

### Распределение ответственности

| Этап | Ответственный |
|------|---------------|
| NEW | OMS |
| CONFIRMED | Менеджер OMS |
| PAID | Менеджер OMS / Financial Engine |
| PREPARING | Менеджер OMS |
| SHIPPED | OMS + API Нова Пошта |
| DELIVERED | API Нова Пошта |
| FUNDS_RECEIVED | OMS + Financial Engine |
| COMPLETED | OMS |

---

### Описание сценария

После оформления заказа OMS автоматически создаёт новый заказ и присваивает ему статус **NEW**.

После проверки заказа менеджером OMS заказ переводится в статус **CONFIRMED**, а покупателю автоматически направляется письмо с подтверждением заказа и инструкциями по оплате.

После подтверждения успешной оплаты заказ получает статус **PAID**.

Менеджер приступает к подготовке заказа, после чего OMS переводит его в статус **PREPARING**.

После передачи отправления перевозчику и получения подтверждения приёма заказ получает статус **SHIPPED**.

OMS автоматически начинает мониторинг доставки посредством API Delivery Provider.

После подтверждения перевозчиком получения отправления покупателем заказ переводится в статус **DELIVERED**.

При использовании наложенного платежа OMS ожидает подтверждения поступления денежных средств.

После получения подтверждения заказ получает статус **FUNDS_RECEIVED**.

После завершения всех обязательств OMS автоматически закрывает жизненный цикл заказа переводом в статус **COMPLETED**.

---

### Контрольные точки

В рамках данного сценария OMS контролирует:

- регистрацию заказа;
- подтверждение заказа;
- подтверждение оплаты;
- подготовку заказа;
- передачу перевозчику;
- подтверждение доставки;
- поступление денежных средств;
- завершение заказа.

---

### Используемые интеграции

Данный сценарий использует следующие подсистемы платформы:

- OMS;
- Workflow Engine;
- Order Events;
- Reminder Engine;
- Delivery Provider API;
- Financial Engine;
- Checkbox (ПРРО);
- Email Engine.

---

### Примечание

Выбор службы доставки осуществляется покупателем при оформлении заказа.

OMS автоматически использует API выбранного Delivery Provider для получения актуальной информации о доступных странах, способах доставки, отделениях, почтоматах, тарифах, ограничениях и сроках доставки.

OMS автоматизирует выполнение выбранного пользователем сценария исполнения заказа, но не принимает решение о выборе службы доставки вместо покупателя.

Статус **MEETING_SCHEDULED** отражает факт согласования встречи и не изменяется при переносе даты, времени или места передачи.

Все изменения параметров встречи фиксируются в журнале событий заказа (Order Events).

Статус **HANDED_OVER** подтверждает факт личной передачи товара покупателю и завершает этап исполнения заказа по данному сценарию.



---


## 5.5 Fulfillment Scenario — Цифровий товар

### Назначение

Настоящий сценарий используется для продажи цифровых товаров, передаваемых покупателю посредством электронных каналов связи.

К цифровым товарам относятся:

- электронные книги;
- цифровые издания;
- иные электронные материалы, распространяемые платформой PRACTICULARIUM.

Сценарий не предусматривает физическую доставку товара.

---

### Используемые статусы

```text
NEW
 ↓
CONFIRMED
 ↓
PAID
 ↓
DIGITAL_DELIVERY
 ↓
COMPLETED
```

---

### Распределение ответственности

| Этап | Ответственный |
|------|---------------|
| NEW | OMS |
| CONFIRMED | Менеджер OMS |
| PAID | Financial Engine |
| DIGITAL_DELIVERY | OMS |
| COMPLETED | OMS |

---

### Описание сценария

После оформления заказа OMS автоматически создаёт новый заказ и присваивает ему статус **NEW**.

После проверки заказа менеджером OMS заказ переводится в статус **CONFIRMED**, а покупателю автоматически направляется подтверждение заказа и инструкции по оплате.

После подтверждения успешной оплаты заказ получает статус **PAID**.

OMS автоматически выполняет передачу цифрового товара покупателю посредством предусмотренного канала доставки.

После успешной передачи заказ получает статус **DIGITAL_DELIVERY**.

После завершения всех обязательств OMS автоматически переводит заказ в статус **COMPLETED**.

---

### Контрольные точки

В рамках данного сценария OMS контролирует:

- регистрацию заказа;
- подтверждение заказа;
- подтверждение оплаты;
- успешную передачу цифрового товара;
- завершение заказа.

Контроль сроков и автоматические напоминания выполняются Reminder Engine в соответствии с утверждёнными бизнес-правилами платформы.

---

### Используемые интеграции

Данный сценарий использует следующие подсистемы платформы:

- OMS;
- Workflow Engine;
- Order Events;
- Reminder Engine;
- Financial Engine;
- Checkbox (ПРРО);
- Email Engine.

Интеграция с Delivery Provider в данном сценарии отсутствует.

---

### Примечание

Передача цифрового товара осуществляется автоматически после выполнения условий, предусмотренных бизнес-процессом платформы.

Способ предоставления доступа к цифровому товару определяется соответствующим модулем платформы и может включать персональные ссылки, защищённые загрузки, личный кабинет пользователя или иные утверждённые механизмы распространения цифрового контента.


---

# 6. Order Events

## Назначение раздела

Настоящий раздел определяет архитектуру журнала событий заказа (Order Events), используемого платформой PRACTICULARIUM.

Order Events являются неотъемлемой частью OMS и предназначены для регистрации всех значимых действий, происходящих в процессе жизненного цикла заказа.

Журнал событий обеспечивает полную историю обработки заказа независимо от его текущего статуса.

---

## Назначение Order Events

Order Events используются для:

- фиксации действий пользователей;
- регистрации автоматических действий системы;
- хранения истории изменения заказа;
- регистрации взаимодействия с внешними сервисами;
- обеспечения аудита операций;
- формирования аналитической информации;
- диагностики ошибок;
- восстановления истории обработки заказа.

---

## Архитектурная роль

Order Events являются самостоятельной подсистемой платформы.

Журнал событий не управляет жизненным циклом заказа.

Изменение статуса заказа может сопровождаться созданием одного или нескольких событий.

В то же время большинство событий не изменяют статус заказа.

---

## Основной принцип

Статус заказа отвечает на вопрос:

> **«В каком состоянии сейчас находится заказ?»**

Order Event отвечает на вопрос:

> **«Что произошло с заказом?»**

Эти понятия являются независимыми и не заменяют друг друга.



---


## Основные принципы

### Один заказ — множество событий

Каждый заказ может содержать неограниченное количество событий.

Количество событий не влияет на количество статусов.

---

### События неизменяемы

После регистрации событие не изменяется и не удаляется.

При необходимости фиксируется новое событие.

История обработки заказа должна оставаться полной и достоверной.

---

### Хронологический порядок

Все события фиксируются в порядке их возникновения.

Каждое событие содержит дату и время регистрации.

При отображении истории заказа события сортируются по времени создания.

---

### Независимость от Workflow

Order Events не определяют допустимые переходы между статусами.

Контроль жизненного цикла заказа выполняется исключительно Workflow Engine.

---

### Независимость от Fulfillment Scenario

Все сценарии исполнения заказа используют единый механизм Order Events.

Тип сценария влияет только на перечень создаваемых событий, но не изменяет архитектуру журнала событий.

---

### Полнота истории

История заказа должна позволять полностью восстановить последовательность обработки заказа без обращения к внешним источникам информации.


---

## Статус и событие

| Статус                            | Событие                                                                                                 |
|---------                          |----------                                                                                               |
| Отражает текущее состояние заказа | Отражает произошедшее действие                                                                          |
| Один активный статус              | Неограниченное количество событий                                                                       |
| Изменяется при переходе Workflow  | Создаётся при любом значимом действии                                                                   |
| Управляет жизненным циклом заказа | Формирует историю заказа                                                                                |
| Используется OMS                  | Используется OMS, Workflow, Reminder Engine, Delivery Provider, Financial Engine и другими подсистемами |


---


## Структура события

Каждое событие Order Events представляет собой отдельную запись журнала истории заказа.

Минимальный состав события включает следующие поля.

| Поле | Назначение |
|------|------------|
| Event ID | Уникальный идентификатор события |
| Order Number | Номер заказа |
| Event Type | Тип события |
| Event Source | Источник возникновения события |
| Event Date | Дата и время события |
| User / System | Пользователь или подсистема, создавшая событие |
| Description | Краткое описание события |
| Metadata | Дополнительные данные события (при необходимости) |

---

### Примечание

Состав полей может быть расширен в будущих версиях платформы без изменения архитектуры журнала событий.

Все дополнительные сведения должны сохраняться в структуре Metadata либо в специализированных расширениях Order Events.


---


## Источники событий

События могут создаваться различными подсистемами платформы.

На текущем этапе предусматриваются следующие источники.

| Источник | Назначение |
|----------|------------|
| OMS | Действия менеджера и внутренние операции системы |
| Workflow Engine | Изменение жизненного цикла заказа |
| Reminder Engine | Автоматические напоминания и контроль сроков |
| Delivery Provider | События служб доставки |
| Financial Engine | Финансовые операции |
| Checkbox | Регистрация фискальных чеков |
| Email Engine | Отправка сообщений |
| Customer Portal *(будущий модуль)* | Действия покупателя |
| CRM Layer *(будущий модуль)* | Постпродажное взаимодействие |

---

### Примечание

Перечень источников событий может расширяться по мере развития платформы.

Добавление нового источника не требует изменения архитектуры Order Events.


---


## Использование Order Events

Журнал событий используется всеми подсистемами платформы PRACTICULARIUM как единый механизм регистрации истории обработки заказа.

Order Events являются основным источником информации для:

- просмотра полной истории заказа;
- внутреннего аудита операций;
- анализа работы сотрудников;
- диагностики ошибок;
- формирования аналитической отчётности;
- контроля исполнения бизнес-процессов;
- восстановления последовательности обработки заказа.

---

### Архитектурное правило

Order Events не изменяют статус заказа самостоятельно.

Изменение статуса заказа может сопровождаться созданием одного или нескольких событий.

При этом наличие нового события не означает обязательного изменения текущего статуса заказа.

---

### Пример

Статус заказа:

```
SHIPPED
```

Во время нахождения заказа в данном статусе могут быть зарегистрированы следующие события:

- создана транспортная накладная;
- отправлено письмо покупателю;
- отправлено SMS-уведомление;
- изменён предполагаемый срок доставки;
- выполнена синхронизация с Delivery Provider;
- зарегистрировано получение новой информации от перевозчика.

Все перечисленные действия являются событиями и не изменяют статус заказа.

---

### Примечание

Подробная архитектура журнала событий, перечень типов событий, структура хранения и правила обработки определяются отдельным документом **ORDER-EVENTS-01**.




---



# 7. Exception Flow

## Назначение раздела

Настоящий раздел определяет общие принципы обработки исключительных ситуаций, возникающих в процессе исполнения заказа.

Исключительными считаются события, которые выводят заказ из стандартного сценария жизненного цикла либо требуют выполнения дополнительных действий, не предусмотренных основным Workflow.

Обработка исключительных ситуаций выполняется в соответствии с отдельными бизнес-регламентами платформы.

---

## Основной принцип

Исключительная ситуация не создаёт новый жизненный цикл заказа.

Она является отклонением от стандартного сценария исполнения.

После устранения причины исключения заказ может:

- вернуться в основной Workflow;
- завершиться исключительным статусом;
- быть окончательно закрыт.

---

## Архитектурный принцип

Все исключительные ситуации должны:

- быть зарегистрированы в Order Events;
- фиксироваться в журнале истории заказа;
- иметь ответственного за обработку;
- не нарушать целостность жизненного цикла заказа.

Подробные правила обработки каждой исключительной ситуации определяются отдельными бизнес-регламентами платформы.


---


## Основные типы исключительных ситуаций

Платформа PRACTICULARIUM предусматривает обработку следующих основных категорий исключительных ситуаций.

| Исключительная ситуация | Назначение |
|--------------------------|------------|
| CANCELLED | Отмена заказа |
| PAYMENT_FAILED | Ошибка или невозможность подтверждения оплаты |
| NOT_RECEIVED | Покупатель не получил заказ |
| RETURNED | Возврат товара отправителю |
| REFUND | Возврат денежных средств покупателю |
| DELIVERY_PROBLEM | Проблемы при доставке |
| DIGITAL_DELIVERY_FAILED | Ошибка передачи цифрового товара |
| SYSTEM_ERROR | Внутренняя ошибка платформы или интеграции |

---

### Примечание

Перечень исключительных ситуаций может быть расширен в будущих версиях платформы.

Полный порядок обработки каждой исключительной ситуации определяется отдельным документом **ORDER-EXCEPTIONS-01**.


---


## Общие правила обработки исключений

При возникновении исключительной ситуации OMS обязана:

1. зарегистрировать событие в Order Events;

2. сохранить полную информацию о причине возникновения исключения;

3. определить ответственную сторону за дальнейшую обработку;

4. уведомить заинтересованных участников процесса (при необходимости);

5. обеспечить возможность последующего анализа возникшей ситуации.

---

### Принцип прослеживаемости

Каждая исключительная ситуация должна быть полностью прослеживаема.

История возникновения, обработки и завершения исключения должна сохраняться в журнале Order Events.

Удаление информации об исключительных ситуациях не допускается.

---

### Принцип повторяемости

Обработка одинаковых исключительных ситуаций должна выполняться по единым бизнес-правилам независимо от выбранного Fulfillment Scenario.

Особенности конкретного сценария исполнения заказа могут влиять только на последовательность действий, но не изменяют общие архитектурные принципы обработки исключений.


---


## Архитектурный принцип

Exception Flow не является самостоятельным жизненным циклом заказа.

Исключительные ситуации представляют собой отклонения от основного Workflow и обрабатываются в соответствии с утверждёнными бизнес-регламентами платформы.

После завершения обработки исключения заказ:

- возвращается в основной Workflow;

или

- завершается соответствующим исключительным статусом;

или

- закрывается в соответствии с установленными правилами обработки.

---

### Примечание

Подробные сценарии обработки исключительных ситуаций определяются отдельным документом **ORDER-EXCEPTIONS-01** и не входят в область действия настоящего документа.



---


# 8. Термины и определения

Настоящий раздел устанавливает единое толкование терминов, используемых в документации PRACTICULARIUM Platform.

Все архитектурные документы платформы используют данные определения как официальные.

---

## Order

Заказ покупателя, зарегистрированный OMS и являющийся объектом управления жизненным циклом платформы.

Каждый заказ имеет уникальный номер, собственный жизненный цикл и историю обработки.

---

## Order Lifecycle

Полный жизненный цикл заказа от момента его регистрации до окончательного завершения либо обработки исключительной ситуации.

Жизненный цикл определяется Workflow Engine.

---

## Status

Текущее состояние заказа в рамках жизненного цикла.

В каждый момент времени заказ может иметь только один активный статус.

---

## Order Event

Отдельное действие или событие, произошедшее во время обработки заказа.

События формируют журнал истории заказа и не заменяют статусы жизненного цикла.

---

## Workflow

Последовательность допустимых переходов между статусами заказа.

Контроль Workflow выполняется Workflow Engine.

---

## Fulfillment Scenario

Утверждённый сценарий исполнения заказа, определяющий последовательность выполнения операций по передаче товара покупателю.

Сценарий выбирается в зависимости от способа исполнения заказа.

---

## Delivery Provider

Внешний сервис доставки, интегрированный с платформой посредством API.

Delivery Provider предоставляет актуальную информацию о доступных странах, способах доставки, тарифах, сроках и иных параметрах доставки.

---

## Reminder Engine

Подсистема платформы, отвечающая за автоматический контроль сроков выполнения бизнес-процессов и отправку соответствующих напоминаний.

---

## Financial Engine

Подсистема платформы, обеспечивающая взаимодействие с платёжными сервисами, банками и финансовыми интеграциями.

---

## Checkbox

Интеграция платформы с программным регистратором расчётных операций (ПРРО), используемым для формирования и регистрации фискальных чеков в соответствии с требованиями законодательства.

---

## OMS (Order Management System)

Центральная подсистема платформы PRACTICULARIUM, обеспечивающая управление заказами, их жизненным циклом и взаимодействием всех связанных модулей.

---

## Customer

Покупатель или иной пользователь платформы, оформивший заказ.

---

## Manager

Уполномоченный сотрудник издательства, осуществляющий обработку заказов посредством OMS.

---

## API

Программный интерфейс взаимодействия между платформой PRACTICULARIUM и внешними сервисами.

---

## Single Source of Truth

# 9. Заключительные положения

## Область применения документа

Настоящий документ определяет архитектурные принципы построения жизненного цикла заказа (Order Lifecycle) платформы PRACTICULARIUM.

Документ является обязательным для использования при разработке, сопровождении и развитии всех подсистем, взаимодействующих с OMS.

Настоящий документ не определяет техническую реализацию отдельных модулей платформы.

## Связь с другими архитектурными документами

Настоящий документ является частью единой архитектурной документации PRACTICULARIUM Platform и используется совместно со следующими документами:

- Architecture Standards;

- Delivery Providers *(будущий документ)*;
- Financial Engine *(будущий документ)*;
- Order Exceptions *(будущий документ)*.

Совместное использование данных документов обеспечивает единое понимание архитектуры платформы всеми участниками проекта.


---


## Принципы развития документа

Настоящий документ определяет архитектуру жизненного цикла заказа и должен оставаться стабильным.

При дальнейшем развитии платформы допускается:

- добавление новых Fulfillment Scenarios;
- добавление новых типов товаров;
- добавление новых Delivery Provider;
- добавление новых способов оплаты;
- добавление новых интеграций;
- расширение перечня исключительных ситуаций.

При этом изменение архитектурных принципов настоящего документа допускается только в случае необходимости пересмотра базовой модели Order Lifecycle.

Изменение отдельных модулей платформы не должно требовать изменения архитектуры жизненного цикла заказа.

---

## Версионирование

Все изменения настоящего документа должны сопровождаться изменением номера версии и регистрацией соответствующих изменений в истории проекта.

Использование неутверждённых редакций документа в качестве основы для разработки не допускается.


---


## Заключение

Архитектура Order Lifecycle, представленная в настоящем документе, определяет единый подход к управлению жизненным циклом заказа в платформе PRACTICULARIUM.

Настоящий документ устанавливает единые принципы использования статусов, сценариев исполнения заказа, журнала событий и обработки исключительных ситуаций.

Принятые архитектурные решения обеспечивают:

- масштабируемость платформы;
- независимость подсистем;
- возможность подключения новых интеграций;
- единообразие обработки заказов;
- поддержку различных способов исполнения и оплаты;
- основу для последующей автоматизации бизнес-процессов и аналитики.

Настоящий документ является одним из базовых архитектурных документов PRACTICULARIUM Platform и используется в качестве основы для разработки Workflow Engine, OMS и связанных подсистем.
